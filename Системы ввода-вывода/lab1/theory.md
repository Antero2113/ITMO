# Теоретические сведения к лабораторной работе №1

## «Принципы организации ввода/вывода без операционной системы»

---

# 1. Архитектура RISC-V

## Что такое архитектура процессора

Архитектура процессора (ISA — Instruction Set Architecture) определяет:

* набор инструкций процессора;
* доступные регистры;
* форматы команд;
* способы обращения к памяти;
* механизм системных вызовов;
* режимы привилегий.

Именно ISA определяет, как программное обеспечение взаимодействует с аппаратной частью компьютера.

---

## Что такое RISC-V

RISC-V — это современная открытая процессорная архитектура типа RISC (Reduced Instruction Set Computer).

Главная идея RISC-подхода — использование небольшого количества простых инструкций вместо большого набора сложных команд.

Особенности RISC-V:

* открытая спецификация;
* отсутствие лицензионных ограничений;
* простая и понятная структура инструкций;
* возможность расширения архитектуры;
* поддержка различных разрядностей (32/64/128 бит).

RISC-V активно используется:

* во встраиваемых системах;
* в исследовательских проектах;
* в разработке операционных систем;
* в образовательных целях.

---

## Архитектура RV32IMAC

В лабораторной работе используется конфигурация:

```text id="v0y5z7"
RV32IMAC
```

Расшифровка:

| Расширение | Назначение                   |
| ---------- | ---------------------------- |
| RV32       | 32-битная архитектура        |
| I          | базовый набор инструкций     |
| M          | операции умножения и деления |
| A          | атомарные операции           |
| C          | compressed-инструкции        |

---

### Compressed instructions

Расширение `C` позволяет использовать сокращенные 16-битные инструкции вместо стандартных 32-битных.

Это уменьшает размер программы и снижает потребление памяти.

---

# 2. Режимы работы процессора RISC-V

В RISC-V существует система уровней привилегий.

Она нужна для разделения:

* пользовательских программ;
* операционной системы;
* низкоуровневой прошивки.

---

## Основные режимы

| Режим  | Назначение                 |
| ------ | -------------------------- |
| U-mode | пользовательские программы |
| S-mode | операционная система       |
| M-mode | машинный режим             |

---

## User Mode (U-mode)

Это минимальный уровень привилегий.

В нем работают обычные приложения.

Ограничения:

* нет прямого доступа к устройствам;
* нельзя управлять памятью процессора;
* нельзя выполнять privileged-инструкции.

---

## Supervisor Mode (S-mode)

В этом режиме обычно работает ядро операционной системы.

ОС получает возможность:

* работать с виртуальной памятью;
* управлять процессами;
* обрабатывать исключения;
* вызывать функции OpenSBI.

---

## Machine Mode (M-mode)

Это самый привилегированный режим процессора.

M-mode имеет полный доступ ко всем аппаратным ресурсам системы.

В этом режиме выполняются:

* загрузчики;
* прошивки;
* OpenSBI;
* обработчики критических исключений.

---

## Почему OpenSBI работает именно в M-mode

Некоторые аппаратные операции могут выполняться только из M-mode:

* управление hart;
* выключение системы;
* настройка прерываний;
* доступ к machine registers.

Поэтому OpenSBI работает как промежуточный слой между ОС и аппаратурой.

---

# 3. Что такое OpenSBI

## SBI — Supervisor Binary Interface

SBI — это стандартный интерфейс взаимодействия между:

* Supervisor software (ядро ОС);
* SEE (Supervisor Execution Environment).

SEE — это низкоуровневое программное окружение, работающее в M-mode.

---

## Зачем нужен SBI

Без SBI каждой операционной системе пришлось бы:

* самостоятельно управлять всеми устройствами;
* реализовывать код для каждой платформы;
* работать напрямую с аппаратурой.

SBI решает эту проблему.

Он предоставляет унифицированный интерфейс доступа к аппаратным функциям.

---

## Что делает OpenSBI

OpenSBI — реализация SBI с открытым исходным кодом.

Основные задачи OpenSBI:

1. Инициализация оборудования.
2. Настройка процессора.
3. Подготовка памяти.
4. Запуск ядра.
5. Предоставление API для ОС.

---

## Аналогии с BIOS и UEFI

OpenSBI по смыслу похож на:

| Технология | Где используется |
| ---------- | ---------------- |
| BIOS       | старые ПК        |
| UEFI       | современные ПК   |
| U-Boot     | embedded Linux   |

Как BIOS подготавливает x86-компьютер к запуску ОС, так OpenSBI подготавливает RISC-V систему.

---

# 4. Процесс загрузки системы

## Последовательность загрузки

При запуске лабораторной работы происходит следующая цепочка:

```text id="uk8k8l"
QEMU → OpenSBI → boot → main()
```

---

## Этапы загрузки

### 1. Запуск QEMU

QEMU создает виртуальную RISC-V систему.

Эмулируются:

* процессор;
* память;
* UART;
* таймеры;
* контроллеры устройств.

---

### 2. Запуск OpenSBI

OpenSBI:

* начинает выполняться в M-mode;
* настраивает оборудование;
* подготавливает процессор;
* загружает ядро по адресу `0x80200000`.

---

### 3. Передача управления ядру

После загрузки OpenSBI передает управление точке входа:

```asm id="m4svd6"
boot
```

---

### 4. Выполнение start.S

Код:

* настраивает стек;
* вызывает функцию `main()`.

---

### 5. Работа основной программы

В `main()` запускается пользовательское меню.

---

# 5. Что такое QEMU

QEMU — программный эмулятор компьютерных систем.

Он позволяет запускать разные архитектуры без физического оборудования.

В лабораторной работе используется:

```bash id="vv0ejz"
qemu-system-riscv32
```

---

## Почему используется QEMU

Использование QEMU позволяет:

* запускать RISC-V программы на обычном ПК;
* отлаживать bare-metal код;
* тестировать низкоуровневые системы;
* работать без реального RISC-V процессора.

---

## Машина virt

Платформа `virt` — стандартная виртуальная RISC-V система в QEMU.

Она содержит:

* RISC-V CPU;
* UART;
* RAM;
* таймеры;
* контроллер прерываний;
* virtio-устройства.

---

# 6. Bare-metal программирование

## Что означает bare-metal

Bare-metal программа работает напрямую на аппаратуре:

* без Linux;
* без Windows;
* без драйверов ОС;
* без стандартной библиотеки libc.

Программа самостоятельно управляет всеми устройствами.

---

## Особенности bare-metal программ

Разработчик должен самостоятельно:

* настраивать стек;
* организовывать вывод;
* работать с памятью;
* реализовывать взаимодействие с устройствами.

---

# 7. Memory-Mapped I/O

## Принцип работы MMIO

Во многих архитектурах устройства отображаются в адресное пространство памяти.

Это называется:

```text id="0xys6g"
Memory-Mapped I/O
```

---

## Как это работает

Устройство получает набор адресов памяти.

Чтение/запись этих адресов приводит к взаимодействию с устройством.

Например:

```c id="d8ym5v"
*(volatile unsigned char*)0x10000000 = 'A';
```

выводит символ через UART.

---

## Почему используется volatile

Ключевое слово:

```c id="4pbv0k"
volatile
```

говорит компилятору:

> значение может изменяться аппаратурой в любой момент.

Без `volatile` компилятор мог бы удалить важные обращения к устройствам при оптимизации.

---

# 8. UART

## Что такое UART

UART (Universal Asynchronous Receiver-Transmitter) — последовательный интерфейс передачи данных.

Используется для:

* консольного ввода;
* вывода текста;
* отладки embedded-систем.

---

## UART в лабораторной работе

В QEMU UART расположен по адресу:

```c id="5qxy4f"
0x10000000
```

---

## Основные регистры UART

| Регистр | Адрес    | Назначение       |
| ------- | -------- | ---------------- |
| THR     | base + 0 | передача символа |
| LSR     | base + 5 | состояние UART   |

---

## Line Status Register (LSR)

Регистр состояния содержит флаги состояния UART.

---

### Бит 0x20

```text id="6p3wzh"
UART готов к передаче
```

Если бит установлен — можно отправлять символ.

---

### Бит 0x01

```text id="u8x3vn"
Получен новый символ
```

Если бит установлен — данные готовы к чтению.

---

# 9. Реализация putchar

## Назначение функции

Функция `uart_putc()` выводит один символ в консоль.

---

## Код функции

```c id="c9v5yz"
void uart_putc(char c)
{
    while (!(UART_LSR & 0x20));
    UART_THR = c;
}
```

---

## Подробный разбор

### Ожидание готовности UART

```c id="4s0snp"
while (!(UART_LSR & 0x20));
```

Программа постоянно проверяет статус UART.

Пока UART занят передачей предыдущего символа — выполнение останавливается.

---

### Отправка символа

```c id="qug93k"
UART_THR = c;
```

Запись байта в THR запускает передачу символа.

---

# 10. Реализация getchar

## Назначение

Получение символа от пользователя.

---

## Код функции

```c id="0fmkme"
char uart_getc()
{
    while (!(UART_LSR & 0x01));
    return UART_THR;
}
```

---

## Подробный разбор

### Ожидание ввода

Программа непрерывно проверяет:

```c id="7k5v4v"
UART_LSR & 0x01
```

Пока пользователь ничего не ввел — программа ожидает.

---

## Почему getchar блокирующая

Функция не возвращает управление до появления символа.

Поэтому Console Getchar считается blocking function.

---

# 11. Инструкция ECALL

## Назначение ECALL

ECALL — механизм системного вызова в RISC-V.

Он используется для перехода между уровнями привилегий.

---

## В лабораторной работе

ECALL используется для обращения к OpenSBI.

---

## Принцип работы

1. Аргументы помещаются в регистры.
2. Выполняется `ecall`.
3. Процессор передает управление OpenSBI.
4. OpenSBI выполняет функцию.
5. Результат возвращается обратно.

---

# 12. Регистры вызова OpenSBI

---

## Extension ID (EID)

Регистр:

```text id="2k7vsh"
a7
```

Содержит идентификатор расширения SBI.

---

## Function ID (FID)

Регистр:

```text id="b2htdf"
a6
```

Определяет конкретную функцию внутри расширения.

---

## Аргументы функции

Регистры:

```text id="j2zv9h"
a0-a5
```

используются для передачи параметров.

---

## Возвращаемые значения

После выполнения ECALL:

| Регистр | Назначение              |
| ------- | ----------------------- |
| a0      | код ошибки / результат  |
| a1      | дополнительное значение |

---

# 13. Что такое hart

## Определение

Hart (Hardware Thread) — аппаратное ядро RISC-V процессора.

Фактически это отдельный поток выполнения инструкций.

---

## Аналогия

Hart можно представить как CPU core.

Если система имеет 4 hart — значит процессор способен выполнять 4 независимых потока инструкций.

---

# 16. Стек

## Что такое стек

Стек — область памяти для:

* локальных переменных;
* адресов возврата;
* временных данных.

---

## Stack Pointer

Регистр:

```text id="cnm86y"
sp
```

хранит адрес вершины стека.

---

## Инициализация стека

В `start.S` выполняется:

```asm id="xtz8s3"
la sp, stack_top
```

После этого функции могут корректно использовать стек.

# 17. Компиляция bare-metal программы

---

## Почему нельзя использовать обычный GCC

Стандартные программы ожидают наличие:

* операционной системы;
* libc;
* системных вызовов.

В bare-metal среде этого нет.

---

## Используемые флаги

### -ffreestanding

Отключает зависимость от стандартной среды выполнения.

---

### -nostdlib

Не подключает стандартные библиотеки.

---

### -march=rv32imac

Указывает архитектуру процессора.

# 19. Общая логика работы программы

1. QEMU запускает виртуальную машину.
2. OpenSBI инициализирует систему.
3. OpenSBI загружает ядро.
4. Выполняется `boot`.
5. Инициализируется стек.
6. Вызывается `main()`.
7. Отображается меню.
8. Пользователь выбирает пункт.
9. Выполняется ECALL.
10. OpenSBI выполняет функцию.
11. Результат выводится через UART.
