Процесс	        Где работает	  Что делает
WAL Sender	    PRIMARY	          Отправляет WAL-сегменты (журнал изменений) репликам
WAL Receiver	STANDBY	          Получает WAL от primary и применяет к своей БД

replicator — это роль (пользователь) для репликации
Строка в pg_hba.conf разрешает этой роли подключаться в режиме репликации
Процесс репликации = WAL Sender (на primary) + WAL Receiver (на standby)
Режим репликации — специальный протокол PostgreSQL для передачи WAL, не для SQL-запросов

вместо:
psql -h pg_b -U postgres -d appdb
# Позволяет: SELECT, INSERT, UPDATE, DELETE, CREATE...

идёт:
psql -h pg_b -U replicator -d replication
# Только для: STREAMING REPLICATION команды, не для SQL

Параметры конфига:
listen_addresses = '*'

PG_PROMOTE

Когда вы вызываете эту функцию на сервере-реплике, внутри PostgreSQL происходит следующий процесс :

Слой 1: Внутренности PostgreSQL (Сигналы и процессы)
Вызов pg_promote(wait_seconds => 60); на сервере-реплике (например, pg_b) запускает цепочку событий внутри PostgreSQL.

Вызов функции: Функция выполняется в сессии postgres (клиентский процесс). Она ничего не делает сама, а просто отправляет сигнал главному процессу сервера.
Отправка сигнала (SIGUSR1): Функция находит PID (идентификатор) главного процесса postmaster и отправляет ему сигнал SIGUSR1.
SIGUSR1 — это пользовательский сигнал в Unix/Linux, зарезервированный для прикладных нужд приложения. PostgreSQL использует его для внутренней координации, в частности, для команды "выйти из режима восстановления".
Реакция postmaster: Главный процесс, получив SIGUSR1, понимает: "Мне приказали стать мастером". Он меняет своё внутреннее состояние.
Завершение recovery: postmaster заставляет процесс startup (который крутился в цикле, применяя WAL от старого мастера) завершиться. На этом этапе сервер больше не является standby.
Активация: Запускаются обычные процессы: checkpointer, bgwriter, walwriter, и главное — открывается возможность записи (сервер больше не находится в режиме read-only).
Ожидание (Wait): Параметр wait_seconds => 60 заставляет вашу SQL-сессию не завершаться сразу. Она в цикле проверяет, вышла ли система из recovery, и ждёт этого до 60 секунд.
Важно: Если wait_seconds => 0 (аналог false), сессия отправит сигнал и сразу вернёт true, не дожидаясь реальной готовности сервера к записи.

Итог по слою 1: Это "кнопка запуска двигателя" внутри одного сервера PostgreSQL.

Слой 2: Роль Pgpool-II в Failover
Pgpool-II не управляет сигналами SIGUSR1. Вместо этого он действует как координатор и наблюдатель.

Что делает Pgpool:
Обнаружение сбоя: Pgpool через механизм health_check (каждые 5 секунд по вашей настройке) пытается подключиться к каждому бэкенду. Если подключение к pg_a (текущий primary) перестало отвечать, Pgpool запускает процедуру failover.
Выбор кандидата: Pgpool анализирует состояние остальных узлов, смотрит на их replication_state и выбирает наиболее подходящего кандидата (например, pg_b с наименьшим replication_delay).
Вызов скрипта: Pgpool НЕ вызывает pg_promote() сам по себе. Он вызывает ваш скрипт (failover_command), который вы прописали в pgpool.conf.

text
failover_command = '/usr/local/bin/pgpool-failover.sh %d %P %m %H %M %N'
Ответственность скрипта: Задача вашего скрипта (который вы показывали ранее) — подключиться к выбранному кандидату (NEW_MAIN_HOST) и выполнить ту самую команду: psql -c "SELECT pg_promote();".
Завершение failover: После того как pg_promote() завершится (и скрипт выйдет), Pgpool переключает внутренний флаг, и теперь все новые запросы на запись будут направляться на новый primary (pg_b).
Итог по слою 2: Pgpool — это "диспетчер", который решает, когда и на каком узле нажать кнопку "promote". Саму кнопку нажимает ваш скрипт, вызывая pg_promote() через psql.

Итоговая схема работы (Ваш кластер)
Падение pg_a: Primary перестаёт отвечать (docker stop).

Pgpool через health_check (порт 5432) перестаёт получать ответ.
Pgpool принимает решение: "Узел 0 (pg_a) умер. Он был primary. Узел 1 (pg_b) - лучший кандидат".
Pgpool выполняет failover_command с параметрами, где %m (ID нового primary) = 1, %H (хост нового primary) = pg_b.
Ваш скрипт (pgpool-failover.sh) на основе этих параметров подключается к pg_b.
Ваш скрипт выполняет: SELECT pg_promote(wait_seconds => 60);.

PostgreSQL на pg_b:

Процесс postmaster получает SIGUSR1 от psql сессии.
Завершает процесс startup (останавливает применение WAL).
Запускается в режиме read-write.
Pgpool видит, что pg_b теперь primary, и начинает направлять на него все операции записи.
Таким образом, сигнал SIGUSR1 — это последний шаг внутри PostgreSQL, который происходит после того, как Pgpool обнаружил сбой, выбрал новый мастер и через ваш скрипт отдал команду на его активацию.
