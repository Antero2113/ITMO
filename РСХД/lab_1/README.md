# Параметры конфигурации PostgreSQL

## Параметры подключения

### `listen_addresses` (string)
Задаёт адреса TCP/IP, по которым сервер будет принимать подключения клиентских приложений. Значение принимает форму списка, разделённого запятыми, из имён и/или числовых IP-адресов узлов.

- `*` — все имеющиеся IP-интерфейсы
- `0.0.0.0` — все адреса IPv4
- `::` — все адреса IPv6
- `''` (пустой список) — только Unix-сокеты

По умолчанию: `localhost` (только локальный интерфейс «замыкания»)

Параметр можно задать только при запуске сервера.

### `port` (integer)
TCP-порт, открываемый сервером. Используется для всех IP-адресов, через которые сервер принимает подключения.

По умолчанию: `5432`

Параметр можно задать только при запуске сервера.

### `max_connections` (integer)
Максимальное число одновременных подключений к серверу БД.

По умолчанию: `100` (может быть меньше, если ядро накладывает ограничения)

Параметр можно задать только при запуске сервера.

**Важно**: PostgreSQL определяет размер некоторых ресурсов в зависимости от значения `max_connections`. Увеличение этого значения приводит к выделению большего объёма этих ресурсов, включая общую память.

Для ведомого сервера значение должно быть больше или равно значению на ведущем.

## Параметры памяти

### `shared_buffers` (integer)
Объём памяти для буферов в разделяемой памяти. Это кэш страниц данных PostgreSQL.

- Минимальное значение: `128kB`
- По умолчанию: `128MB`
- Рекомендация для выделенного сервера с RAM ≥ 1 ГБ: 25% от ОЗУ
- Максимальная рекомендация: не более 40% ОЗУ

Параметр можно задать только при запуске сервера.

### `temp_buffers` (integer)
Максимальный объём памяти для временных буферов в каждом сеансе. Используются для работы с временными таблицами.

По умолчанию: `8MB`

Параметр можно изменить в сеансе до первого обращения к временным таблицам.

### `work_mem` (integer)
Базовый максимальный объём памяти для внутренних операций при обработке запросов (сортировка, хеш-таблицы). При превышении используются временные файлы на диске.

По умолчанию: `4MB`

**Важно**: В сложных запросах может использоваться несколько операций одновременно, и они могут выполняться параллельно в разных сеансах, поэтому общий расход памяти может многократно превышать `work_mem`.

### `effective_cache_size` (integer)
Оценка планировщиком эффективного размера дискового кэша, доступного для одного запроса. Влияет на выбор между индексным и последовательным сканированием.

- По умолчанию: `4GB`
- Не выделяет реальную память, только подсказка для планировщика

## Параметры WAL и надёжности

### `fsync` (boolean)
Обеспечивает физическую запись изменений на диск через системные вызовы `fsync()`.

- `on` (по умолчанию) — гарантия восстановления после сбоя
- `off` — выше производительность, но риск неисправимой порчи данных при сбое

Отключать рекомендуется только если данные можно легко восстановить из внешнего источника.

### `commit_delay` (integer)
Пауза перед сохранением WAL. Позволяет сгруппировать несколько транзакций в одну операцию записи.

По умолчанию: `0` (задержка отсутствует)

Добавляется только если активны как минимум `commit_siblings` других транзакций.

### `commit_siblings` (integer)
Минимальное число одновременно открытых транзакций, при котором будет добавляться задержка `commit_delay`.

По умолчанию: `5`

## Параметры контрольных точек

### `checkpoint_timeout` (integer)
Максимальное время между автоматическими контрольными точками в WAL.

- Диапазон: от `30s` до `1d`
- По умолчанию: `5min`

Увеличение параметра может увеличить время восстановления после сбоя.

---

## Краткое назначение параметров

| Параметр | Назначение |
|----------|------------|
| `listen_addresses` | какие сетевые интерфейсы слушать |
| `port` | порт для TCP-подключений |
| `max_connections` | максимум одновременных подключений |
| `shared_buffers` | размер кэша страниц данных PostgreSQL |
| `temp_buffers` | память под временные таблицы в сеансе |
| `work_mem` | память под сортировку/хеш-таблицы в запросе |
| `effective_cache_size` | оценка размера системного кэша для планировщика |
| `fsync` | гарантировать ли физическую запись на диск |
| `commit_delay` | задержка перед записью WAL для группировки транзакций |
| `commit_siblings` | минимальное число транзакций для активации commit_delay |
| `checkpoint_timeout` | интервал между автоматическими контрольными точками |


## Способы запуска и остановки сервера PostgreSQL на FreeBSD

### Запуск сервера

#### 1. **pg_ctl** (основной способ)
```bash
pg_ctl -D /var/db/postgres2/zfb92 start
pg_ctl -D /var/db/postgres2/zfb92 -l /var/db/postgres2/zfb92/log/postgresql.log start
```
- Основная утилита для управления сервером
- Требует указания директории данных (`-D`)
- Можно указать файл для лога (`-l`)

#### 2. **service** (если PostgreSQL установлен как порт/пакет)
```bash
sudo service postgresql start
```
- Используется, если PostgreSQL установлен из портов или пакетов
- Управляется через rc-скрипты FreeBSD

#### 3. **/usr/local/etc/rc.d/postgresql** (напрямую)
```bash
sudo /usr/local/etc/rc.d/postgresql start
```

### Остановка сервера

#### 1. **pg_ctl stop** (основной способ)
```bash
pg_ctl -D /var/db/postgres2/zfb92 stop -m <режим>
```

**Режимы остановки:**
- **`smart`** (`-m smart`) — ждёт завершения всех клиентских подключений
- **`fast`** (`-m fast`) — откатывает активные транзакции, разрывает соединения (режим по умолчанию)
- **`immediate`** (`-m immediate`) — аварийное завершение без корректного завершения транзакций, требует восстановления при следующем запуске

#### 2. **service** (остановка сервиса)
```bash
sudo service postgresql stop
```

#### 3. **/usr/local/etc/rc.d/postgresql** (напрямую)
```bash
sudo /usr/local/etc/rc.d/postgresql stop
```

## Отличия способов запуска/остановки на FreeBSD

| Аспект | pg_ctl | service/rc.d |
|--------|--------|--------------|
| **Требует указания -D** | Да | Нет |
| **Управление конкретным кластером** | Да | Управляет сервисом в целом |
| **Автозапуск при загрузке** | Нет | Да (если включено в rc.conf) |
| **Требует прав root** | Нет (от пользователя postgres) | Да |
| **Гибкость режимов остановки** | Полный контроль (-m) | Определяется настройками сервиса |

## Отличия режимов остановки

| Режим | Поведение | Когда использовать |
|-------|-----------|-------------------|
| **smart** | Ждёт завершения всех клиентских подключений | Плановое завершение, не срочно |
| **fast** | Откатывает активные транзакции, обрывает соединения | Обычное завершение (по умолчанию) |
| **immediate** | Аварийное завершение без корректного завершения транзакций | Только если сервер не отвечает |
